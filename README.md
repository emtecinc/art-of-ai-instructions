# art-of-ai-instructions

Centralized instruction intelligence for AI-assisted Playwright test automation and manual test generation.

## What This Is

A shared instruction repository providing reusable framework rules, coding patterns, utility guidance, and AI-readable discovery for Salesforce Playwright test automation. Consuming projects reference this repo to ensure consistent, high-quality AI-generated code across all project instances.

## Architecture: Centralized vs Local

### Centralized (this repo — mounted as `ai-instructions/` in consuming projects)

| What | Mounted Path in Consuming Project | Why |
|---|---|---|
| `instructions/*.instructions.md` | `ai-instructions/instructions/*.instructions.md` | Framework rules auto-loaded via `applyTo` — single authority per concern |
| `utility/*.md` | `ai-instructions/utility/*.md` | Helper utility reference docs — read on demand |
| `agents/manual-test-step-generator.agent.md` | `ai-instructions/agents/manual-test-step-generator.agent.md` | Standalone text-only agent — no MCP or browser dependency |
| `README.md`, `INDEX.md`, `AGENTS.md` | `ai-instructions/README.md`, etc. | Discovery, indexing, agent registry |
| `templates/workflows/` | `ai-instructions/templates/workflows/` | CI/CD templates for consuming projects |

### Local (consuming project)

| What | Why |
|---|---|
| `.github/copilot-instructions.md` | Project-specific context (org URL, app names, object list) |
| `.github/local-override.md` | Project-level rule overrides (use sparingly) |
| `.github/agents/playwright-test-planner.agent.md` | Requires local MCP server + live browser |
| `.github/agents/playwright-test-generator.agent.md` | Requires local MCP server + live browser |
| `.github/agents/playwright-test-healer.agent.md` | Requires local MCP server + live browser |
| Source code: `pages/`, `workflows/`, `tests/`, `test-data/` | Generated by agents |
| `.env` | Credentials and environment variables |

### Why Playwright Agents Are Local Only

The planner, generator, and healer agents depend on:
- **MCP tool server** (`playwright-test`) for browser automation
- **Live browser** for step-by-step UI verification
- **Local workspace** for file paths, env vars, installed Playwright browsers
- **Project execution environment** for running and debugging tests

These cannot be centralized. They MUST live in `.github/agents/` in each consuming project.

The **Manual Test Step Generator** is the only centralized agent because it is text-only, uses only `read/edit/search` tools, and has zero runtime dependency.

### Instruction Loading Flow

1. Agent edits a file under `pages/` → `ai-instructions/instructions/page-objects.instructions.md` auto-loads via `applyTo: "pages/**/*.ts"`
2. Agent edits under `workflows/` → `ai-instructions/instructions/workflows.instructions.md` auto-loads
3. Agent edits under `tests/` → `ai-instructions/instructions/spec-files.instructions.md` + `ai-instructions/instructions/salesforce-stability.instructions.md` auto-load
4. Agent edits under `test-data/` → `ai-instructions/instructions/test-data.instructions.md` auto-loads

### Instruction Precedence

| Priority | Source | Scope |
|---|---|---|
| 1 (highest) | `ai-instructions/instructions/*.instructions.md` | Reusable framework rules — always active, auto-loaded |
| 2 | `.github/copilot-instructions.md` | Project-specific context (org URL, objects, apps) |
| 3 | `.github/local-override.md` | Project-level exceptions — only overrides the specific behavior it explicitly mentions |

**Rule:** Centralized instructions are the default authority. Local files add context or override specific behaviors — they never replace the full instruction set.

## 3-Layer Test Architecture

All Playwright agents enforce a strict separation of concerns:

| Layer | Location | Responsibility |
|---|---|---|
| **Page Object** | `pages/<object>/` | Locators, element interactions, `expect()` assertions |
| **Workflow** | `workflows/<object>/` | Business-level step orchestration via `this.step()` |
| **Spec** | `tests/<object>/` | Declarative test orchestration — zero DOM, zero assertions |

## Agent Pipeline

```
Planner ──(plan)──► Generator ──(code)──► Healer
```

| Agent | Purpose | Location |
|---|---|---|
| **Planner** | Explores live app, designs test scenarios, saves plans to `specs/` | Local |
| **Generator** | Implements plans as Playwright code following 3-layer architecture | Local |
| **Healer** | Debugs and fixes failing tests without violating framework rules | Local |
| **Manual Test Step Generator** | Converts Playwright codegen output to structured manual test cases | **Centralized** |

See [AGENTS.md](AGENTS.md) for detailed agent boundaries and dependencies.

## How to Set Up a Consuming Project

### 1. Reference this repo

Add as a submodule or symlink so `instructions/`, `utility/`, `agents/`, and `templates/` are accessible under `ai-instructions/`:

```bash
git submodule add <repo-url> ai-instructions
```

This makes all centralized files accessible at `ai-instructions/instructions/`, `ai-instructions/utility/`, etc.

### 2. Create local files

In `.github/`:
- **`copilot-instructions.md`** — project context (org URL, app names, object list, custom behaviors)
- **`local-override.md`** — rule overrides (use sparingly — most rules should stay centralized)

In `.github/agents/`:
- **`playwright-test-planner.agent.md`**
- **`playwright-test-generator.agent.md`**
- **`playwright-test-healer.agent.md`**

### 3. Use agents

In GitHub Copilot Chat:
- `@playwright-test-planner` — create test plans
- `@playwright-test-generator` — generate test code from plans
- `@playwright-test-healer` — debug failing tests
- `@AI-test-case-step-generator` — produce manual test cases from codegen output

### 4. Copy CI/CD templates

Copy from `templates/workflows/` to `.github/workflows/` and customize.
